# 项目再次审查（2026-03-05）

## 审查范围
- 会话链路：API -> Orchestrator -> Planner -> Executor -> Renderer
- 技能发现与脚本执行
- 可靠性（重试、幂等）
- 错误治理与可观测
- 测试覆盖度

## 当前已具备能力
1. 四层主链路可运行并有测试覆盖（包含 API 入口）。
2. 技能发现机制为 Claude 风格：`SKILL.md` 发现 + frontmatter 解析 + markdown 聚合。
3. 脚本执行包含 action 白名单、路径边界校验、超时控制。
4. 新增全局 trace 透传和统一异常返回 schema。
5. Executor 支持重试与退避（可配置）。
6. API 增加了幂等能力（支持 `idempotency_key`）。

## 本轮新增修正
1. **幂等缓存容量治理**：将无限增长的进程内缓存改为可配置上限 + FIFO 驱逐（默认 1000）。
2. **补充回归测试**：新增缓存驱逐测试，确保不会无限增长。

## 仍建议继续推进的事项（按优先级）
### P1
- 将幂等存储从进程内迁移到 Redis（支持多副本部署）。
- 增加执行队列与并发上限，防止高峰挤压。
- 增加 planner schema 强校验（JSON Schema / pydantic）并细化错误码。

### P2
- 脚本执行隔离：资源配额（CPU/内存）+ 权限沙箱。
- 增加审计日志（输入摘要/输出摘要/耗时/错误码）。
- 真实 LLM + 真实脚本的冒烟回放集，接入 CI。

## 结论
项目已从“Demo 可跑”提升到“具备基础生产治理能力”，但在**分布式幂等、执行隔离、可观测指标体系**上仍需继续投入。
